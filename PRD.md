# Product Requirements Document — task-context-hub

> Trello 카드 단위로 연결되는 Task Context Document(TCD)를 통해 작업 맥락 · 설계 결정 · AI 협업 기록을 한 곳에 모아 사람과 AI가 작업을 끊김 없이 이어갈 수 있도록 돕는 개발팀용 컨텍스트 레이어 도구의 제품 요구사항 문서.

본 PRD는 [`project-context.md`](./project-context.md)를 입력으로 작성되며, 두 문서가 충돌할 경우 PRD 변경 → project-context.md 반영 순으로 동기화한다.

---

## 1. Executive Summary

task-context-hub은 Trello로 작업 카드/상태를 관리하는 소규모~중규모 개발팀(2~15명)을 위한 **task 단위 컨텍스트 레이어**다. Trello 카드 1개에 1:1로 연결되는 **Task Context Document(TCD)** 를 두고, 작업 배경 · 요구사항 · Decision Log · AI Work Log · 관련 코드/PR/문서 링크 · Handoff Summary를 구조화된 마크다운으로 누적한다.

핵심 차별 기능은 **AI Context Package 생성**이다. 버튼 한 번으로 현재 task에 대한 모든 컨텍스트(목적/진행 상태/요구사항/설계 결정/관련 파일/주의사항/다음 요청)를 단일 텍스트로 생성하여, 사용자가 Claude / Cursor / ChatGPT / Codex 등 어떤 AI에도 복사 한 번으로 컨텍스트를 이식할 수 있게 한다.

MVP는 웹 데스크톱 우선 반응형으로 출시한다. Trello/Claude만 외부 연동하며, GitHub/Slack 자동 동기화 · Activity Timeline · Context Quality Score · 과거 유사 작업 추천은 Phase 2 이상으로 미룬다. 본 PRD는 MVP 범위의 기능 요구사항(FR), 비기능 요구사항(NFR), 도메인 요구사항, 기술적 제약을 정의한다.

## 2. Success Criteria (SMART)

본 절의 성공 지표는 **S**pecific · **M**easurable · **A**chievable · **R**elevant · **T**ime-bound 기준을 따른다. 측정 기간은 MVP 출시 후 3개월(이하 *MVP+3M*) 시점이며, 측정 대상은 MVP를 도입한 파일럿 팀이다.

| ID | 지표 | 목표값 | 측정 방법 | 측정 시점 |
| --- | --- | --- | --- | --- |
| SC-01 | 인수인계 파악 시간 단축 | Handoff Summary 사용 시 다른 개발자가 작업을 파악하는 데 걸리는 평균 시간이 **50% 이상 단축** | 동일 task에 대해 (a) TCD 없이 인수인계 받은 그룹과 (b) Handoff Summary 사용 그룹의 첫 코드 변경까지 걸린 시간을 비교 (사내 측정 폼) | MVP+3M |
| SC-02 | AI 재설명 시간 단축 | AI Context Package 사용 시 AI에게 컨텍스트를 재설명하는 데 걸리는 평균 시간 **70% 이상 단축** | Context Package 사용 전/후의 "AI 첫 응답까지 걸린 시간" 비교 (사용자 자가 보고 + 표본 측정) | MVP+3M |
| SC-03 | 컨텍스트 작성률 | MVP 사용 팀의 **70% 이상**이 task당 최소 1개 이상의 Decision Log 또는 AI Work Log를 작성 | 워크스페이스별 TCD 대비 (DecisionLog ∪ AIWorkLog) 보유 비율 집계 | MVP+3M |
| SC-04 | 리뷰 전 TCD 참조율 | 리뷰어가 PR 리뷰 전 Task Context Document를 참조하는 비율 **60% 이상** | TCD 페이지 뷰 이벤트 중 "PR 링크 클릭 직전" 또는 "리뷰 시작 24시간 이내" 비율 측정 | MVP+3M |
| SC-05 | TCD 생성 성공률 | Trello 카드 URL/ID 입력 → TCD 자동 생성 성공률 **95% 이상** (네트워크/권한 오류 제외) | 자동 생성 API 호출 대비 성공 응답 비율 (서버 로그) | MVP+1M |
| SC-06 | Context Package 생성 응답 시간 | AI Context Package 생성 요청의 **p95 응답 시간 ≤ 10초** | API 응답 시간 메트릭 p95 | MVP+1M |

## 3. Product Scope

### 3.1 MVP

MVP는 다음 기능을 포함한다. 모든 기능은 §5의 FR로 측정 가능한 기준을 동반한다.

- **회원가입/로그인** (이메일 + JWT)
- **워크스페이스 + Trello 연동** (워크스페이스 단위 멤버 · 토큰 관리)
- **Trello 카드 → TCD 자동 생성** (카드 URL/ID 입력 → 카드 메타데이터 가져와 TCD 1개 생성)
- **Task Context Page** (작업 개요/목적/배경/요구사항/관련 파일/진행 상태/남은 작업/이슈)
- **Decision Log** (결정 / 검토 대안 / 선택 이유 / 영향 범위)
- **수동 ResourceLink** (Repo/Branch/Commit/PR/Doc/API 링크 등록)
- **AI Work Log** (요약 중심: 도구/목적/요청 요약/결과 요약/적용 여부/관련 파일/후속 작업)
- **AI Context Package 생성** (단일 텍스트/마크다운, 클립보드 복사)
- **Handoff Summary** (현재 상태/완료/미완료/주의사항/먼저 봐야 할 파일/다음 추천 작업)
- **컨텍스트 상태 관리** (작업 상태와 별개의 컨텍스트 상태 6단계)

### 3.2 Growth (Phase 2)

- Timeline / Activity History (task별 변경 이력 시간순)
- Context Quality Score (충실도 점수 + 부족 항목 표시)
- Git / PR 자동 연동 (GitHub/GitLab)
- Slack / 회의록 / 문서 링크 자동 연동
- 다국어 UI 확장(영문 등)

### 3.3 Vision (Phase 3 이후)

- 과거 유사 작업 추천(임베딩 기반 검색)
- AI 도구별 직접 API 통합 기반 자동 AI Work Log 수집
- 팀 단위 지식 그래프 / 도메인 지식 베이스
- 엔터프라이즈용 SSO/SCIM, 감사로그, 데이터 거버넌스

### 3.4 명시적 제외 (Out of Scope)

본 제품은 다음을 **목표로 하지 않는다**.

- Trello를 **대체**하는 작업 카드/상태/칸반 도구 (Trello는 task 정의 도구로 계속 사용)
- Git/PR **자동** 동기화 (MVP는 수동 링크로 핵심 가치 먼저 검증)
- Slack/회의록 **자동** 연동 (Phase 2)
- AI 도구 직접 API 통합 기반 **자동** AI Work Log (후순위)
- 네이티브 모바일 앱 (데스크톱 웹 반응형으로 커버)
- 코드 호스팅/리뷰 UI (GitHub 등에 위임)
- 작업 시간 트래킹/캘린더 일정 관리(범위 밖)

## 4. User Journeys

본 절은 MVP에서 검증해야 하는 핵심 사용자 여정을 기술한다. 각 단계는 §5의 FR과 매핑된다.

### 4.1 J-01 신규 사용자 온보딩 → Trello 연동

1. 사용자는 이메일 + 비밀번호로 회원가입한다. *(FR-01)*
2. 첫 로그인 후 워크스페이스를 1개 생성하고 워크스페이스 owner가 된다. *(FR-04)*
3. Trello API Key/Token을 등록하여 워크스페이스를 Trello에 연결한다. *(FR-05)*
4. Trello 보드 목록이 워크스페이스에 표시되어, 사용자는 어떤 보드를 동기화할지 인지할 수 있다. *(FR-06)*

### 4.2 J-02 Trello 카드 → Task Context Document 생성

1. 사용자는 Trello 카드 URL을 복사하여 task-context-hub에 붙여넣는다. *(FR-07)*
2. 시스템이 카드 메타데이터(제목/담당자/상태/라벨/마감일)를 가져와 TCD 1개를 자동 생성한다. *(FR-08)*
3. 사용자는 자동 생성된 TCD에 진입하여 목적/배경/요구사항을 작성한다. *(FR-09)*
4. 사용자는 TCD의 컨텍스트 상태를 "요구사항 정리됨"으로 변경한다. *(FR-14)*

### 4.3 J-03 설계 결정 · AI 작업 기록

1. 사용자는 설계 토론 후 Decision Log를 작성한다(결정/대안/이유/영향 범위). *(FR-10)*
2. 사용자는 Claude / Cursor 등으로 작업하다가 결과 요약을 AI Work Log로 기록한다(도구/목적/요청 요약/결과 요약/적용 여부/관련 파일). *(FR-11)*
3. 사용자는 관련 PR · 변경 파일 · 설계 문서 링크를 ResourceLink로 등록한다. *(FR-12)*

### 4.4 J-04 AI Context Package로 작업 이어가기

1. 사용자는 TCD 화면의 "Generate AI Context Package" 버튼을 누른다. *(FR-13)*
2. 시스템이 작업명/목적/진행 상태/요구사항/설계 결정/관련 파일/완료·남은 작업/주의사항/다음 요청을 단일 마크다운 텍스트로 생성한다. *(FR-13)*
3. 사용자는 1회 클릭으로 클립보드 복사하고, Claude / Cursor / ChatGPT 등 임의 AI 도구에 붙여넣어 작업을 이어간다. *(FR-13)*

### 4.5 J-05 인수인계 흐름

1. 작업 중단 또는 담당자 변경 시, 사용자는 Handoff Summary를 작성한다(현재 상태/완료/미완료/주의사항/먼저 봐야 할 파일/다음 추천 작업). *(FR-15)*
2. 인수받는 사용자는 TCD + Handoff Summary를 읽고, 필요 시 AI Context Package를 생성해 AI 도움을 받는다. *(FR-13, FR-15)*
3. 인수받은 사용자는 컨텍스트 상태를 "인수인계 가능 → 구현 진행 중"으로 변경한다. *(FR-14)*

### 4.6 J-06 PR 리뷰 전 컨텍스트 참조

1. 리뷰어는 PR 본문의 TCD 링크 또는 task-context-hub 검색을 통해 TCD에 진입한다.
2. 리뷰어는 목적/배경/Decision Log를 읽어 설계 의도를 파악한 뒤 코드 리뷰를 시작한다.

## 5. Functional Requirements

본 PRD의 모든 FR은 다음 규칙을 따른다.

- "사용자가 ~할 수 있다" 형식으로 기술한다.
- 각 FR은 **측정 가능한 테스트 기준(AC, Acceptance Criteria)** 을 동반한다.
- 구현 세부(라이브러리명/스키마 컬럼명/프레임워크 내부 동작)는 기술하지 않는다.
- 모호한 형용사("빠른", "직관적인")는 사용하지 않고, 수량/시간/조건으로 표현한다.

### FR-01 회원가입

- **요구사항**: 사용자는 이메일과 비밀번호로 계정을 생성할 수 있다.
- **AC**:
  - 사용자가 유효한 이메일(RFC 5322) + 8자 이상 비밀번호 입력 시 5초 이내에 가입이 완료된다.
  - 동일 이메일로의 중복 가입 시도는 거부되며 사용자에게 사유가 표시된다.
  - 가입 직후 사용자는 별도 로그인 없이 인증 상태가 된다.

### FR-02 로그인

- **요구사항**: 사용자는 이메일/비밀번호로 로그인할 수 있다.
- **AC**:
  - 올바른 자격증명 입력 시 2초 이내에 인증 토큰을 발급받고 워크스페이스 진입 화면으로 이동한다.
  - 잘못된 자격증명은 동일한 일반 오류 메시지를 반환한다(이메일/비밀번호 어느 쪽이 틀렸는지 노출하지 않는다).
  - 5회 연속 실패 시 해당 계정에 대해 15분간 로그인 시도를 제한한다.

### FR-03 로그아웃 / 세션 만료

- **요구사항**: 사용자는 로그아웃하거나 세션 만료 시 자동으로 비인증 상태로 돌아갈 수 있다.
- **AC**:
  - 로그아웃 즉시 해당 토큰으로의 보호된 API 호출이 401을 반환한다.
  - 세션 토큰의 만료 시각이 명시되어 있으며 만료 후의 호출은 401을 반환한다.

### FR-04 워크스페이스 생성/멤버 관리

- **요구사항**: 사용자는 워크스페이스를 생성하고 다른 사용자를 멤버로 초대할 수 있다.
- **AC**:
  - 워크스페이스 생성자는 OWNER 역할을 가진다.
  - OWNER는 이메일로 다른 사용자를 MEMBER로 추가/제거할 수 있다.
  - MEMBER는 해당 워크스페이스에 속한 TCD/Log/Link만 조회 및 편집할 수 있고, 다른 워크스페이스의 데이터에는 접근할 수 없다.

### FR-05 Trello 연동 등록

- **요구사항**: OWNER는 워크스페이스에 Trello API Key/Token(또는 OAuth)을 등록하여 Trello와 연결할 수 있다.
- **AC**:
  - 등록 즉시 토큰 유효성 검증을 수행하고, 실패 시 사용자에게 사유가 표시된다.
  - 등록된 토큰은 평문으로 저장되지 않으며, 클라이언트에 다시 반환되지 않는다.
  - 등록된 통합은 해제(disconnect)할 수 있으며, 해제 후에는 즉시 외부 API 호출이 중단된다.

### FR-06 Trello 보드 목록 조회

- **요구사항**: 사용자는 워크스페이스에 연결된 Trello 계정의 보드 목록을 볼 수 있다.
- **AC**:
  - 보드 목록은 카드의 동기화 대상 선택 UI에서 사용 가능하다.
  - 200개 이하의 보드는 5초 이내에 표시된다.

### FR-07 Trello 카드 URL/ID 입력

- **요구사항**: 사용자는 Trello 카드 URL 또는 카드 ID를 입력하여 해당 카드를 TCD에 연결할 수 있다.
- **AC**:
  - 유효한 카드 URL/ID 입력 시 카드 식별이 성공한다.
  - 동일 카드에 대해 이미 TCD가 존재하면 새로 생성하지 않고 기존 TCD로 이동한다.

### FR-08 TCD 자동 생성

- **요구사항**: 사용자는 Trello 카드 URL/ID 입력으로 카드 메타데이터를 가져와 TCD 1개를 자동 생성할 수 있다.
- **AC**:
  - 자동 생성된 TCD는 다음 필드를 카드에서 가져와 초기화한다: 제목, 담당자(있는 경우), 상태, 라벨, 마감일.
  - 생성된 TCD의 컨텍스트 상태 초기값은 "미작성"이다.
  - 자동 생성 응답 시간은 p95 ≤ 5초이다.
  - Trello API 호출 실패 시 사용자에게 재시도 가능한 형태로 오류 메시지가 표시된다.

### FR-09 Task Context Page 편집

- **요구사항**: 사용자는 TCD의 본문 섹션(작업 개요/목적/배경/요구사항/관련 도메인·모듈/관련 파일/진행 상태/남은 작업/이슈·주의사항)을 작성/수정할 수 있다.
- **AC**:
  - 본문은 마크다운으로 입력하고 미리보기에서 동일하게 렌더링된다.
  - 각 섹션은 `## Level 2` 마크다운 헤더로 식별 가능하며, 헤더 텍스트는 시스템 약속에 따라 고정된다.
  - 저장 후 1초 이내에 다른 멤버가 새로고침 시 변경 사항을 볼 수 있다.

### FR-10 Decision Log 작성/조회

- **요구사항**: 사용자는 TCD에 Decision Log를 추가하고 누적된 로그를 시간순으로 볼 수 있다.
- **AC**:
  - Decision Log는 결정/검토 대안/선택 이유/영향 범위 4개 항목 모두를 입력해야 저장된다.
  - 동일 TCD의 Decision Log는 생성 시각 내림차순으로 정렬된다.
  - 본인이 작성한 Decision Log는 수정/삭제 가능하며, 다른 사용자의 로그는 OWNER만 삭제할 수 있다.

### FR-11 AI Work Log 작성/조회

- **요구사항**: 사용자는 TCD에 AI Work Log를 요약 형태로 추가하고 누적 로그를 볼 수 있다.
- **AC**:
  - AI Work Log는 사용 도구, 목적, 입력 요청 요약, 결과 요약, 적용 여부 5개 필드를 필수로 한다.
  - 관련 파일 목록과 후속 작업 메모는 선택 입력이다.
  - 입력 요청 요약과 결과 요약은 각각 최대 4,000자로 제한되며 초과 시 저장이 거부된다.

### FR-12 ResourceLink 등록/조회

- **요구사항**: 사용자는 TCD에 관련 코드/PR/문서 링크를 종류(REPO/BRANCH/COMMIT/PR/DOC/API)와 함께 등록하고 목록으로 볼 수 있다.
- **AC**:
  - URL은 `http://` 또는 `https://`로 시작해야 저장된다.
  - 종류와 제목은 필수, 설명은 선택이다.
  - 등록된 링크는 종류별로 그룹핑되어 표시되며, 클릭 시 새 탭으로 열린다.

### FR-13 AI Context Package 생성

- **요구사항**: 사용자는 TCD 화면에서 단일 버튼으로 현재 task의 AI Context Package(텍스트/마크다운)를 생성하고 클립보드에 복사할 수 있다.
- **AC**:
  - 생성된 텍스트는 작업명, 목적, 진행 상태, 요구사항, 설계 결정(요약), 관련 파일, 완료/남은 작업, 주의사항, 다음 요청을 모두 포함한다.
  - 생성 요청의 p95 응답 시간은 10초 이하이다(SC-06).
  - 생성된 결과는 1회 클릭으로 클립보드 복사가 가능하다.
  - 동일 TCD가 변경되지 않은 상태에서 5분 이내 재요청 시 캐시된 결과를 반환한다.

### FR-14 컨텍스트 상태 관리

- **요구사항**: 사용자는 TCD의 컨텍스트 상태를 다음 6단계 중 하나로 명시할 수 있다: 미작성 / 요구사항 정리됨 / 설계 확정됨 / 구현 진행 중 / 인수인계 가능 / 완료 요약됨.
- **AC**:
  - 컨텍스트 상태는 작업 상태(TODO/DOING/REVIEW/DONE)와 별도로 저장된다.
  - 워크스페이스 내 TCD 목록은 컨텍스트 상태별로 필터링이 가능하다.

### FR-15 Handoff Summary 작성

- **요구사항**: 사용자는 TCD에 Handoff Summary를 작성하여 인수인계에 필요한 정보를 한 곳에 모을 수 있다.
- **AC**:
  - Handoff Summary는 현재 상태/완료/미완료/주의사항/먼저 봐야 할 파일/다음 추천 작업 6개 항목을 입력 받는다.
  - 최신 1건만 메인 카드로 표시되며, 과거 Handoff는 이력으로 접근 가능하다.

### FR-16 TCD 검색/목록

- **요구사항**: 사용자는 워크스페이스 내 TCD를 제목/Trello 카드 ID 기준으로 검색하고 목록을 조회할 수 있다.
- **AC**:
  - 제목 검색은 부분 일치를 지원하며 결과는 2초 이내에 표시된다(TCD 5,000건 이하 워크스페이스 기준).
  - 목록은 컨텍스트 상태 / 작업 상태 / 담당자 필터를 지원한다.

### FR-17 권한 가드

- **요구사항**: 사용자는 본인이 속한 워크스페이스의 데이터에만 접근할 수 있다.
- **AC**:
  - 다른 워크스페이스의 TCD/Log/Link에 대한 직접 API 호출은 모두 403을 반환한다.
  - 응답에는 다른 워크스페이스의 리소스 ID 또는 메타데이터가 포함되지 않는다.

## 6. Non-Functional Requirements

본 PRD의 모든 NFR은 다음 템플릿을 따른다.

> **시스템은 [지표]를 [조건]에서 [측정 방법]으로 달성한다.**

### NFR-01 응답 시간(API)

- 시스템은 **단순 조회 API의 p95 응답 시간 ≤ 500ms** 를 동시 접속 사용자 100명 이하 조건에서 **API Gateway 응답 시간 메트릭(p95)** 로 달성한다.

### NFR-02 응답 시간(AI Context Package)

- 시스템은 **Context Package 생성 API의 p95 응답 시간 ≤ 10초** 를 동시 생성 요청 10건 이하 조건에서 **API 응답 시간 메트릭(p95)** 로 달성한다.

### NFR-03 가용성

- 시스템은 **월간 가용성 ≥ 99.5%** 를 유지보수 시간 제외 조건에서 **uptime 모니터링 도구의 월간 가용성 리포트** 로 달성한다.

### NFR-04 데이터 내구성

- 시스템은 **저장된 TCD/Log/Link의 데이터 유실율 = 0** 을 정상 운영 조건에서 **일 1회의 PostgreSQL 백업 및 월 1회 복구 리허설** 로 달성한다.

### NFR-05 보안(인증)

- 시스템은 **인증 토큰 평문 노출 0건** 을 클라이언트-서버 통신 전 구간 조건에서 **HTTPS 강제 + 응답 본문/로그의 토큰 마스킹 점검** 으로 달성한다.

### NFR-06 보안(외부 연동 토큰)

- 시스템은 **외부 연동 토큰의 DB 평문 저장 0건** 을 모든 워크스페이스 조건에서 **DB 컬럼 암호화 + 정기 시크릿 스캔 도구 결과 0건** 으로 달성한다.

### NFR-07 비밀번호 저장

- 시스템은 **비밀번호 평문/가역 저장 0건** 을 모든 사용자 조건에서 **bcrypt 등 단방향 해시 + 정기 DB 스키마/코드 감사** 로 달성한다.

### NFR-08 접근성

- 시스템은 **WCAG 2.1 AA 자동 점검 위반 0건** 을 주요 페이지(랜딩/대시보드/TCD 상세) 조건에서 **axe-core 또는 동등 자동 검증 도구의 위반 보고서** 로 달성한다.

### NFR-09 브라우저 호환성

- 시스템은 **최신 2개 메이저 버전의 Chrome/Edge/Safari/Firefox에서 모든 MVP 기능 정상 동작** 을 데스크톱 환경 조건에서 **수동/자동 크로스 브라우저 회귀 테스트** 로 달성한다.

### NFR-10 반응형

- 시스템은 **너비 1280px 이상에서 모든 화면이 가로 스크롤 없이 표시** 를 데스크톱 우선 조건에서 **수동/자동 반응형 테스트** 로 달성한다.

### NFR-11 관측성

- 시스템은 **모든 5xx 응답에 대한 로그/알림 발생률 100%** 를 운영 환경 조건에서 **중앙 로그 시스템 + 알림 채널 점검** 으로 달성한다.

### NFR-12 외부 API 비용

- 시스템은 **Claude API 월간 비용이 예산 상한 내** 를 정상 사용 패턴 조건에서 **월간 사용량 메트릭 + 예산 알림** 으로 달성한다. (구체적인 상한 금액은 운영 시점에 별도 합의)

### NFR-13 Trello API rate limit

- 시스템은 **Trello API 429 응답 비율 < 1%** 를 평균 부하 조건에서 **외부 API 호출 메트릭 + 백오프/캐시 정책 점검** 으로 달성한다.

### NFR-14 데이터 격리

- 시스템은 **타 워크스페이스 데이터 노출 0건** 을 모든 API 응답 조건에서 **권한 가드 단위/통합 테스트 + 정기 권한 회귀 테스트** 로 달성한다.

### NFR-15 변경 추적 / 감사

- 시스템은 **TCD/Log/Link의 생성/수정/삭제 이벤트 기록률 100%** 를 운영 환경 조건에서 **ActivityEvent 적재 비율 점검** 으로 달성한다.

## 7. Domain Requirements

본 절은 도메인 개념과 그 간의 관계를 명세한다. 컬럼 타입/인덱스 등 물리 스키마는 구현 단계(`apps/api/prisma/schema.prisma`)에서 확정한다.

### 7.1 핵심 도메인 개념

| 도메인 | 정의 |
| --- | --- |
| **User** | task-context-hub 사용자. 이메일/비밀번호로 가입하며, 1명 이상의 워크스페이스에 속한다. |
| **Workspace** | 팀 단위의 협업 공간. Trello 통합을 1개 가질 수 있다. |
| **WorkspaceMember** | User와 Workspace의 N:M 관계. 역할은 OWNER/MEMBER 두 가지. |
| **Integration** | 워크스페이스가 외부 서비스(Trello/GitHub/Slack)와 연결되는 자격. 토큰은 암호화 저장. |
| **TaskContext (TCD)** | Trello 카드 1개와 1:1로 연결되는 컨텍스트 문서. 작업 상태와 컨텍스트 상태를 독립적으로 가진다. |
| **DecisionLog** | TCD에 종속되는 의사결정 기록. (결정, 대안, 이유, 영향 범위) |
| **AIWorkLog** | TCD에 종속되는 AI 협업 기록. (도구, 목적, 요청 요약, 결과 요약, 적용 여부, 관련 파일, 후속 작업) |
| **ResourceLink** | TCD에 종속되는 외부 리소스 링크. 종류(REPO/BRANCH/COMMIT/PR/DOC/API) 구분. |
| **HandoffSummary** | TCD에 종속되는 인수인계 요약. 시간순으로 누적되며 최신본을 메인 카드로 표시. |
| **ActivityEvent** | TCD에 종속되는 변경 이력 이벤트. 모든 도메인 변경을 시간순으로 누적. |
| **AIContextPackage** | TCD를 입력으로 생성된 단일 마크다운 텍스트 산출물. 생성 시점/생성자/내용 기록. |

### 7.2 핵심 도메인 규칙

- **DR-01 — 카드 ↔ TCD 1:1**: 동일 Trello 카드에 대해 워크스페이스당 TCD는 최대 1개만 존재한다.
- **DR-02 — 작업 상태와 컨텍스트 상태의 분리**: TCD의 작업 상태(TODO/DOING/REVIEW/DONE)는 Trello 카드와 동기화되거나 수동 동기화될 수 있지만, **컨텍스트 상태(미작성/요구사항 정리됨/설계 확정됨/구현 진행 중/인수인계 가능/완료 요약됨)** 와는 독립적으로 변경 가능하다.
- **DR-03 — 워크스페이스 데이터 격리**: 모든 도메인 객체는 워크스페이스에 직간접적으로 속하며, 다른 워크스페이스에서 접근/수정/삭제할 수 없다.
- **DR-04 — Decision Log 필수 필드**: 결정/대안/이유/영향 범위 4개 항목을 모두 채워야 저장된다.
- **DR-05 — AI Work Log 필수 필드**: 도구/목적/요청 요약/결과 요약/적용 여부 5개 항목을 필수로 한다.
- **DR-06 — Handoff Summary 단일 최신본**: TCD당 최신 Handoff Summary 1건이 "현재 인수인계 카드"로 표시되며, 과거본은 이력으로 보존된다.
- **DR-07 — ActivityEvent 적재**: 모든 도메인 객체의 생성/수정/삭제는 ActivityEvent로 기록된다.
- **DR-08 — AI Context Package 캐시 무효화**: 해당 TCD/Decision Log/AI Work Log/Handoff Summary/ResourceLink 중 어느 하나라도 변경되면 캐시된 AI Context Package는 무효화된다.

## 8. Technical Constraints

본 절은 PRD가 구현 단계에 부여하는 기술적 제약을 정의한다. 구체적인 라이브러리 선택은 §[`project-context.md`](./project-context.md)에 위임한다.

### 8.1 외부 의존성

- **TC-01 — Trello API rate limit**: 동일 카드 메타데이터는 단일 워크스페이스 단위로 캐시하며, 캐시 미스 외에는 외부 호출을 발생시키지 않는다. 429 발생 시 지수 백오프로 재시도한다.
- **TC-02 — Claude API 비용 통제**: AI Context Package 생성 시 (a) 입력 컨텍스트 길이 상한, (b) 동일 입력에 대한 결과 캐시, (c) 프롬프트 캐싱 적용을 통해 단위 task당 평균 호출 비용을 제어한다.
- **TC-03 — 외부 토큰 암호화**: Trello/GitHub/Slack 토큰은 애플리케이션 레벨 암호화로 저장되며, 키 관리는 시크릿 매니저(또는 환경변수 기반 키)로 분리한다.

### 8.2 보안 / 컴플라이언스

- **TC-04 — 비밀번호 단방향 해시**: 사용자 비밀번호는 단방향 해시(예: bcrypt) 로 저장하며, 평문/가역 저장을 금지한다.
- **TC-05 — 토큰 평문 노출 금지**: 응답 본문/로그/에러 메시지에 인증 토큰 및 외부 통합 토큰의 평문이 포함되지 않는다.
- **TC-06 — AI Work Log 민감 데이터 가드**: AI Work Log는 전체 대화가 아닌 요약 저장을 원칙으로 하며, 시크릿 패턴이 감지되면 저장 전 사용자 검수 단계를 요구한다.

### 8.3 데이터 / 문서 구조

- **TC-07 — 문서 구조 일관성**: TCD 본문은 마크다운 `## Level 2` 헤더 기반의 약속된 섹션 구조를 따른다. 사람과 AI가 동일한 헤더 기준으로 파싱할 수 있도록 헤더 텍스트는 고정된다.
- **TC-08 — Prisma 단일 출처**: 도메인 객체의 스키마는 `apps/api` 측 Prisma 스키마 1개를 단일 출처로 한다.

### 8.4 인프라 / 배포

- **TC-09 — 모노레포 빌드**: `apps/web`과 `apps/api`는 Turborepo 파이프라인 안에서 빌드/테스트되며, 각 앱은 독립적으로 배포 가능해야 한다.
- **TC-10 — 단일 서버 시작 / 확장 경로**: MVP는 단일 web/단일 api 인스턴스로 시작한다. 무거운 통합 동기화(폴링/배치)는 추후 별도 워커/큐로 분리할 수 있는 형태로 구현한다.
- **TC-11 — 로컬 환경**: 로컬 PostgreSQL은 `docker-compose.yml`로 띄울 수 있어야 하며, 별도 설치 없이 개발 환경이 구동 가능하다.

### 8.5 범위 / 비기능

- **TC-12 — 데스크톱 우선**: MVP는 너비 1280px 이상의 데스크톱 우선 반응형으로 설계한다. 네이티브 모바일 앱은 범위 밖이다.

- **TC-14 — 브라우저 호환성**: 최신 2개 메이저 버전의 Chrome/Edge/Safari/Firefox에서 정상 동작해야 한다.
